# From Template To Project

This repository already boots as a working starter, but it still contains template branding, placeholder environment values, and starter flows. Use this checklist to turn it into your actual product before you treat it like a production app.

## 1. Boot The Template Locally First

Run the template once before changing architecture or UI. That gives you a known-good baseline.

```bash
npm install
npm run setup
npm run db:up
npm run dev
```

Then confirm these routes load:

- `/`
- `/login`
- `/register`
- `/dashboard`
- `/robots.txt`
- `/sitemap.xml`

Notes:

- `npm run setup` creates or updates `.env` from `.env.example` and generates a local `BETTER_AUTH_SECRET` automatically.
- `npm run db:up` uses `compose.yaml` to start a local MongoDB container.
- If you already run MongoDB elsewhere, skip `npm run db:up` and point `MONGODB_URI` at your own instance.

## 2. Replace Template Branding

The template still ships with the Foundry Stack identity. Replace that before building new features.

- Update project name, short name, creator, tagline, keywords, navigation, and footer links in `lib/config/site.ts`.
- Replace homepage messaging and calls to action in `app/page.tsx`.
- Update shared shell copy in `components/layout/site-header.tsx` and `components/layout/site-footer.tsx` if the defaults no longer fit.
- Replace generated social assets in `app/opengraph-image.tsx`, `app/twitter-image.tsx`, and `app/icon.svg`.
- Change any leftover starter language in `app/login/page.tsx`, `app/register/page.tsx`, and `app/dashboard/page.tsx`.

## 3. Set Real Environment Values

These are the environment variables that matter for a real project:

| Variable | Local starter value | What to do before launch |
| --- | --- | --- |
| `NEXT_PUBLIC_APP_URL` | `http://localhost:3000` | Set this to your real HTTPS app URL. |
| `MONGODB_URI` | `mongodb://127.0.0.1:27017` | Point this at your persistent MongoDB deployment. |
| `MONGODB_DB_NAME` | `foundry_stack` | Rename this to your app database name. |
| `BETTER_AUTH_SECRET` | generated by `npm run setup` | Set a stable secret in every deployed environment. |
| `BETTER_AUTH_URL` | `http://localhost:3000` | Match this to the public auth base URL. |
| `NEXT_PUBLIC_GA_ID` | `G-XXXXXXXXXX` | Replace with a real GA4 measurement ID or leave blank if you are not using GA yet. |
| `GOOGLE_SITE_VERIFICATION` | blank | Add this when you connect Google Search Console. |

Important:

- The placeholder GA value `G-XXXXXXXXXX` is intentionally ignored by the app. Analytics will not load until you replace it with a real measurement ID.
- Do not rely on the development fallback secret in `auth.ts` outside local setup.

## 4. Finalize Authentication

Authentication is already wired, but you still need to shape it around your real app.

- The main Better Auth config lives in `auth.ts`.
- The route handler is mounted at `app/api/auth/[...all]/route.ts`.
- Server-side session helpers live in `lib/backend/auth/session.ts`.
- Client auth calls live in `lib/auth/client.ts`, `components/auth/login-form.tsx`, `components/auth/register-form.tsx`, and `components/auth/sign-out-button.tsx`.

Before launch, decide whether you need any of the following and update `auth.ts` accordingly:

- additional auth providers
- custom session fields
- stricter trusted origins
- invite-only signup logic
- email verification or password reset flows

At minimum, verify that register, login, dashboard access, and sign-out work against a real MongoDB instance.

## 5. Replace Starter Data Models With Your Own

The backend structure is organized, but it is still a starting point.

- Add your Mongoose schemas under `lib/backend/mongoose/schemas/`.
- Keep connection and database helpers under `lib/backend/mongoose/` and `lib/backend/mongodb/`.
- Add app-specific query helpers under `lib/tanstack/queries/`.
- Build your server routes, server actions, or services around your domain logic instead of keeping everything in pages.
- Replace the starter dashboard in `app/dashboard/page.tsx` with your real authenticated entry point.

If your app needs user profile records beyond Better Auth tables, define that model early and create it during onboarding.

## 6. Finish SEO And Analytics

The template ships with strong SEO defaults, but you still need to customize them.

- Review site-wide metadata in `app/layout.tsx`.
- Review helper logic in `lib/seo/metadata.ts`.
- Update homepage structured data in `app/page.tsx`.
- Expand `app/sitemap.ts` as you add public routes.
- Keep `app/robots.ts` aligned with what should and should not be indexed.

Current behavior:

- `/` is intended to be indexable.
- `/login`, `/register`, and `/dashboard` are intentionally `noindex`.
- Google Analytics loads only when `NEXT_PUBLIC_GA_ID` is a real GA4 measurement ID.

If you want auth routes indexed, change that intentionally. Do not do it by accident.

## 7. Make Deployment Real

The Docker Mongo setup is for local development. Do not treat it as your production database plan.

Before launch:

- choose your hosting platform
- provision a persistent MongoDB deployment
- set all environment variables in that platform
- point `NEXT_PUBLIC_APP_URL` and `BETTER_AUTH_URL` at the deployed domain
- replace every remaining placeholder string, image, and SEO field

If you use multiple environments, keep separate values for local, preview, and production.

## 8. Run The Final Validation Pass

Before calling the project ready, run:

```bash
npm run lint
npm run build
npm run auth:info
```

Then manually verify:

- new user registration works
- login redirects correctly
- protected routes reject unauthenticated users
- sign-out clears the session
- `/robots.txt` and `/sitemap.xml` match your public site
- analytics only loads in environments where you provided a real GA ID

## 9. Definition Of Done

You have converted the template into a real project when all of the following are true:

- the app no longer refers to Foundry Stack
- branding, copy, and navigation match your product
- auth works against a persistent database
- your domain models exist under `lib/backend/`
- SEO metadata, canonical URLs, and social previews match your domain
- analytics and verification env vars are either configured correctly or intentionally disabled
- `npm run lint` and `npm run build` pass cleanly

At that point, the template has stopped being a starter and started being your application.
